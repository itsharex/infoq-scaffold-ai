# 设计 Intelligence 覆盖层（解耦）

本文档是协议技能的可选配套层。
它被有意拆分出来，以保证协议主文档专注于流程门禁。

## 目的

当视觉方向不明确、且你在实现前需要一份紧凑的设计建议包时，使用此覆盖层。

此覆盖层提供：

- 面向产品语境的风格方向建议
- 语义化颜色 token 建议
- 字体搭配建议
- 反模式约束
- 最小化交付前检查项

## 何时使用

仅在 Phase 1 且满足以下任一条件时使用：

- 用户提出“更高级 / 更现代 / 更精致”等抽象视觉目标
- 存在多个可行风格方向，需要显式选型
- 团队需要在静态 Demo 前先统一风格与 token 基线

以下场景可跳过本覆盖层：

- 目标工作区已有足够严格且可直接复用的设计 token 与样式规则
- 任务是无视觉方向变更的小型缺陷修复

## 输入契约

仅收集最小必要输入：

1. 产品类型（如 SaaS dashboard、fintech app、healthcare portal）
2. 页面类型（如 landing、admin 列表/详情、settings、checkout）
3. 用户感知关键词（如 trustworthy、playful、calm、dense、premium）
4. 平台上下文（React admin、Vue admin、weapp React、weapp Vue）
5. 约束条件（如 a11y 等级、性能敏感度、品牌要求）

## 输出契约

输出必须是紧凑建议包，并严格包含以下章节：

1. 风格方向：
   - 一个主风格
   - 一个备选风格
   - 各自一句简短理由
2. 语义色 token：
   - `primary`、`secondary`、`accent/cta`、`bg`、`surface`、`text`、`border`、`error`、`success`
   - 仅在 dark mode 在范围内时再给出明暗双套
3. 字体：
   - 标题字体与正文字体
   - 基础字号与层级步进
4. 交互规则：
   - hover/focus/disabled/loading 基线行为
5. 反模式：
   - 针对此上下文，列出 3-5 条明确“禁止项”
6. 验证钩子：
   - 与该页面绑定的 a11y 与响应式检查点

## 优先推理顺序

为保证输出稳定且节省 token，按以下顺序推理：

1. 可访问性与可读性
2. 交互清晰度与触达安全
3. 布局与响应式行为
4. 风格与品牌表达
5. 动效与精修细节

## 领域提示（精简）

- SaaS/B2B：优先信息清晰、层级明确、强调色克制、数据可读性高
- Fintech：优先可信感与稳健对比，避免过于炫目的渐变
- Healthcare：优先低噪声、可读性强、情绪平稳
- E-commerce：优先 CTA 显著性、可扫读性、商品卡一致性
- Content/Editorial：优先字距行距节奏、行长控制、留白节奏
- Creative/Portfolio：可允许更强风格表达，但导航预测性必须稳定

## 常见反模式

- 同一页面混用多套视觉语言且缺乏主次
- 在组件中散落使用原始颜色值，缺少语义 token
- 缺失焦点态与键盘可操作性
- 响应式仅在单一视口可用
- 装饰性动画遮蔽了状态变化

## 集成规则

此覆盖层不能替代协议门禁，必须继续执行：

- 静态 Demo 前获得 `LAYOUT APPROVED`
- 正式实现前获得 `DEMO APPROVED`
